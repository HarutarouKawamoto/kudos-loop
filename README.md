# kudos-loop
チームの心理的安全性を高めることを目的とした「褒めちぎり駆動開発」支援ツール
「コードレビューが殺伐とする」「感謝を伝えるタイミングがない」といったエンジニアチームの課題を、称賛（Kudos）の循環によって解決するためのWebアプリケーションです。
## 1. プロジェクト概要
開発背景と解決したい課題
エンジニアの生産性には、人とのかかわりがある以上、「心理的安全性」も大事です。本プロジェクトは、日々の小さな貢献を「バッジ」と「コメント」で可視化し、チーム内にポジティブなフィードバックのループを生み出すことを目的としています。コンセプト：PDD (Praise-Driven Development)「褒められて伸びる」を仕組み化し、技術的な成長とチームの結束を同時に促進します。
## 2. 技術スタックと選定理由

| カテゴリー | 選定技術 | 比較と決定理由 |
| :--- | :--- | :--- |
| **フロントエンド** | Next.js (App Router) | BackendとAPIで分離したSPA構成にするため。将来的な拡張性と、TypeScriptでのエンドポイント連携の親和性を重視しました。 |
| **バックエンド** | NestJS (TypeScript) | Expressと比較し、DI（依存性の注入）やモジュール設計が標準化されているため。「疎結合・高凝集」な設計を実践するために採用しました。 |
| **データベース** | PostgreSQL | ユーザーと称賛データの整合性（ACID特性）を保証するため。基本情報の知識を活かし、第三正規化まで厳密に設計しています。 |
| **インフラ** | Docker / Docker Compose | 「環境による差異」を排除し、開発チーム全体で即座に同一の動作環境を構築できる再現性を重視しました。 |
| **CI/CD** | GitHub Actions | コードの品質担保のため。プッシュ時にESLintによる静的解析とJestによる自動テストを強制するパイプラインを構築しました。 |

users : ユーザー情報<br>
badges : 称賛の種類（「ナイスリファクタ」「爆速実装」等）<br>
kudos_transactions : 誰が・誰に・どのバッジを贈ったかの多対多の中間テーブル<br>
バックエンド設計のこだわりアーキテクチャ: Controller / Service / Repository の3層レイヤーを採用し、ビジネスロジックの関心事を分離。<br>
セキュリティ: パスワードは bcrypt でハッシュ化して保存。認証は Passport.js を用いたJWT（JSON Web Token）方式を採用し、ステートレスなスケーラビリティを確保。<br>

## 4. 主要機能
認証・認可機能: ログイン機能および、管理者のみが新しいバッジを作成できる権限管理。<br>
Kudos贈呈機能: ユーザー選択、バッジ選択、コメント入力を伴うトランザクション処理。<br>
リアルタイム・フィード: チーム内の称賛履歴をタイムライン形式で表示。<br>
マイページ（統計）: 自分が獲得したバッジの統計をグラフで可視化。<br>
